📜 The Allegro Systems Manifesto

Core thesis: most things should be static; runtime should emerge from necessity; performance is correctness; durability is baseline.
DOC • doctrine

I. The Web Drifted

The web began as documents, then became applications, then became JavaScript runtimes pretending to be operating systems. We confused complexity with power, hydration with interactivity, orchestration with architecture, and scale with design.

II. Static Is the Default State of Truth

Static is not an optimization. Static is a correctness model.

  • If runtime features aren't used, output must be static.
  • Dynamic behavior must justify its existence.

III. Runtime Is a Capability, Not a Mode

No SSR/static toggles. Runtime activates automatically when features require it: persistence, authentication, personalization, or server-side side effects.

IV. Performance Is a Form of Honesty

Slow systems are lying about work, allocations, state, and abstraction leakage. Allegro treats performance regressions as correctness failures.

V. Progressive Enhancement Is Civil Engineering

HTML must work fully. CSS improves structure. JS enhances interaction. Runtime enables durable side effects.

VI. Durability Is Not Optional

When runtime activates, resilience becomes mandatory engineering, not a library choice. Timeouts, bounded retries with backoff, idempotency keys, circuit breakers, rate limiting, structured errors — these are baseline requirements, not configuration.

VII. JSON Over the Wire Is Pragmatism

JSON is the default wire format. It is universally readable by browsers and integrators without decoding steps and preserves the no-JS baseline. Protocol Buffer support may be added as a future opt-in for teams needing binary efficiency — the default must never compromise progressive enhancement.

VIII. ScoreDB Is Behaviour-Driven, Not Implementation-Driven

SQLite is durable, deterministic, and operationally trivial. Allegro chooses it to avoid premature complexity escalation. Critically, ScoreDB is a behaviour-driven interface: the underlying storage engine is an implementation detail. If PostgreSQL or another driver is added later, the API surface remains identical. You program to ScoreDB, not to SQLite.

IX. Hosting Must Not Capture the Framework

Score builds. Stage hosts. Stage must remain Score-independent at the infrastructure layer to avoid ecosystem monolith-lock.

X. Dependency Governance Is Design Hygiene

Every dependency is an architectural decision. Hidden dependencies are silent policy.

XI. Niche Is Not the Danger

The danger is not being niche — it is accidentally drifting toward pleasing everyone. Allegro is about building scalable, efficient systems in a disciplined way. Self-selection is a feature, not a limitation.

XII. Enforced Recommendations, Permitted Deviations

The supported path is documented and enforced. Deviations are permitted — the escape hatch exists — but the support level drops the moment you leave it. This mirrors how Swift handles unsafe APIs: you know when you've crossed the line.

XIII. The Real Test

The real test will be whether Allegro's declarative APIs make complex systems feel boring to build. If that succeeds, everything else follows.

XIV. The System Is the Product

Allegro is a systems thesis: determinism, bounded behavior, typed contracts, decoupled hosting.

XV. The Strange Thing

The weird part is how un-radical this is. It's just disciplined.

XVI. The Long-Term Bet

Swift can be a first-class web language without inheriting the chaos of JS-heavy stacks—if philosophy is enforced by defaults.

XVII. The Real Innovation: Intent Compiled, Execution Generic

Score's static-first output is a consequence, not the thesis. The actual innovation is this:

Developers express intent. Score compiles intent. A generic interpreter executes it.

HTML encodes structure. CSS encodes appearance. data-as-* attributes encode interaction intent. The JS runtime is a dumb interpreter of that encoded intent — it never executes application logic, only the locked vocabulary Score emitted. This is closer to a declarative VM than a framework.

This is why the system feels cohesive: there is no gap between "what the developer wrote" and "what runs in the browser." The Swift source is the source of truth; the runtime is a read-only execution engine for the compiled artefact. The source is never shipped. The intent is.

This architecture also explains why the mutation boundary is non-negotiable. A VM with an arbitrary instruction set is not a VM — it is an interpreter with unlimited scope. The locked vocabulary is not a limitation; it is the definition of the machine.

⚡ Design Philosophy

Allegro's own standards • Not constraints imposed on Score users

PHI • principles
Scope: This section describes how Allegro's own products — allegro.tld, Libretto, and all first-party interfaces — are designed and built. These are not rules imposed on Score users. Score and Stage place no constraints on how you design your own applications. That said, every component described here ships in ScoreUI — the defaults are calibrated to make this quality level the path of least resistance.

Signs of Taste in Web UI

These are not guidelines. They are the difference between a product that feels made by people who care and one that doesn't.

SignalWhat it means in Score
Every interaction happens in ≤100ms Optimistic UI is the default. Local state updates immediately; network follows. No interaction waits for a round-trip before feeling responsive. The @Action system enforces this — pure synchronous state changes only.
No product tours Score ships no onboarding overlay, coach mark, or tooltip carousel component. If a UI needs explaining, the UI is wrong. Write clearer copy instead.

Persistent, resumable state Local-first by default. Page state persists across reloads via the local store. A user mid-form doesn't lose work on a network hiccup or accidental tab close.
All navigation is ≤3 steps Score's router enforces shallow path depth as a lint rule. Paths deeper than three segments produce a compiler warning. Deep hierarchies are a sign of wrong information architecture, not missing components.
Copyable SVG logo + brandkit The score init CLI generates SVG logo assets in the correct formats alongside app scaffolding. Not a PNG export — actual editable SVG with a locked brandkit structure.
Skeleton loading states Skeleton is a first-class ScoreUI component and the default loading state for any @Query-backed list. Score emits skeleton markup server-side so loading states exist before JS executes.
Copy-paste from clipboard The .copyToClipboard() modifier is one line. Any element can become a copy trigger. Score handles the async Clipboard API and execCommand fallback automatically.
Larger hit targets for buttons and inputs ScoreUI's base button and input styles use min-block-size: 44px; min-inline-size: 44px — Apple's minimum touch target recommendation. The hit target is never smaller than the visible element.
Honest one-click cancel Every destructive action in ScoreUI's confirmation pattern shows one button that actually cancels, with no dark-pattern secondary confirm. The Dialog component's method="dialog" form closes without JS.
⌘K command palette ⌘K opens the command palette on all Allegro properties. On the docs site it drives search across all content and surfaces useful actions — jump to a section, copy a code example, switch theme. The CommandPalette ScoreUI component ships with a complete implementation: keyboard shortcut registration, fuzzy search over registered actions, and a Score-generated action registry from the app's route and action declarations.
Minimal tooltips Score's Tooltip component renders one line of text only. No rich HTML inside tooltips. If you need more than one sentence, you need a different component.
Copy is active voice, max 7 words per sentence ScoreUI's built-in copy — button labels, empty states, error messages — is audited against this rule before each release. No passive voice. No sentences over 7 words.
Optical alignment over geometric Score's spacing scale uses a 4px base unit. Icons are sized to feel optically centred in their containers, not mathematically centred. The .opticalAlign() modifier applies a measured offset when geometric centering produces visible imbalance.
Optimised for left-to-right reading Score's layout primitives default to LTR reading flow. All CSS is emitted as logical properties, so RTL layout is automatic when dir="rtl" is set — no override required. Score sets dir automatically from the active locale.
Reassurance about loss Any form or input that can lose state shows a confirmation when navigating away. Score detects dirty form state automatically from the local store's outbox — if there are pending unsync'd writes, a beforeunload guard is emitted. No manual wiring.

Corollary: What Score Will Not Build For You

Cookie consent banners. Infinite scroll as a default. Autoplay video. Sticky headers that obscure content. Full-page loading spinners. Modal popups on page load. These are not missing features — they are deliberate omissions. Score has no built-in cookie consent component, no autoplay API, no full-page spinner. If you need them, build them. Score won't make them easy on purpose.

📘 Allegro Ecosystem — Product Requirements

Score (framework) • Stage (hosting) • Libretto (first-party app)

PRD • ecosystem

1. Overview

Allegro is a Swift-first web ecosystem built around three components: Score (the framework), Stage (the hosting platform), and Libretto (the first-party reference application that dogfoods both). Each has a clear responsibility and a hard boundary — Score builds, Stage hosts, Libretto validates.

2. Vision

Swift becomes a first-class web platform language. You write Swift, ship static by default, gain durability automatically when runtime is required, and deploy via Stage or anywhere else. Allegro is not trying to win the web — it is trying to define a correct way to build Swift web systems and let people self-select into it.

3. Core Principles

Static-first
Output is static unless runtime features are used. Static is a correctness model, not an optimisation.
Automatic runtime detection
No manual SSR/static toggles. Score detects the minimum runtime required from the features used.
Progressive enhancement
HTML is fully usable. CSS enhances layout. JS adds optional interactivity. Runtime enables durable side effects.
Performance is correctness
Minimal allocations, streaming where possible, no hydration by default, cache-friendly output. Regressions are treated as correctness failures.
Durable by default
When runtime activates: mandatory timeouts, bounded retries, idempotency protection, circuit breakers, rate limiting, structured errors.
Niche is a feature
Self-selection is intentional. Allegro is not about convenience; it is about building scalable, efficient systems in a disciplined way.

4. Component Responsibilities

Score
The Swift web framework. Generates static HTML/CSS by default; activates JS and backend runtime only when features require them. Covers routing, components, styling, interactivity, data, auth, and asset pipeline.
Stage
Swift-native hosting platform. Supports static outputs, runtime binaries, Score apps, and generic Swift executables. Framework-agnostic at its infrastructure core.
Libretto
The first-party reference application. Dogfoods Score and Stage together, validates durability primitives in production, and serves as living documentation of the ecosystem working end-to-end.

Colour Controls

Colour inputs use a hue/chroma/lightness (OKLCH) picker rather than an RGB hex input. This is a deliberate choice — OKLCH is the colour space Score's token system uses, and it produces perceptually uniform palettes. The picker shows:

  • A hue ring
  • A chroma slider (0 = grey, max = vivid)
  • A lightness slider
  • A hex input as a secondary entry point — accepts any valid CSS colour and converts to OKLCH

Semantic roles editable in the builder: accent, surface, text, border, brand (custom role with the full shade scale generated automatically by locking chroma and hue and varying lightness across the scale). Dark mode variants for each role are editable in a toggled dark-mode view of the same controls — the preview page switches to dark when toggled.

Shade generation: When a brand hue is picked, Score generates the full 50–950 shade scale automatically by fixing chroma and hue and varying lightness at perceptually even intervals. The developer can manually adjust individual shades but the automatic generation is good enough for most cases.

6. Boundary Rules

  • Score builds applications. Stage hosts applications. Stage must not depend on Score internals.
  • Stage must not be implemented as a Score application — core infrastructure orchestration stays Score-independent.
  • Stage marketing, docs, and dashboard UI may use Score. This is the only permitted Score dependency in Stage.
  • Libretto must use Score + Stage exclusively — no workarounds, no unsupported paths.

7. Dependency Policy

Dependencies across all Allegro components are restricted to four curated Swift Package Index collections: Apple, Swift.org, SSWG, and Swift Server Community. Any dependency outside these collections requires explicit approval with a written risk/benefit justification. No hidden dependencies are permitted anywhere in the ecosystem.

8. Success Criteria

  • Static sites require no runtime infrastructure
  • Runtime durability prevents the most common distributed system failures by default
  • JSON APIs are first-class, consistent, and browser-native
  • Stage can host Score and non-Score Swift workloads
  • Libretto dogfoods durability primitives in production

📗 Allegro Score — Product Requirements

Static-first Swift web framework • durable runtime • JSON APIs • SQLite persistence

PRD • framework

1. Overview

Score is a Swift web framework. You write Swift; Score emits HTML, CSS, and — only when your code requires it — JS or a backend binary. No toggles. The compiler detects what the code needs and emits exactly that.

2. Syntax (User-Facing)

This section walks through the full API surface in implementation order: Application → Theme → Metadata → Pages → Controllers → Components → Styling → Interactivity → Data. Each concept builds on the last.

Application

Application is the @main entry type. It registers pages, controllers, a theme, and global metadata. There are no manual static/SSR switches — Score detects the minimal runtime required from what the code uses.

@main
struct App: Application {
  var theme      = Theme()         // covered next
  var metadata   = Metadata(...)   // covered after theme
  var pages:       [Page]       { Home(); About() }
  var controllers: [Controller] { AuthController(); PostController() }
}
Result builders: The pages and controllers computed properties use @ArrayBuilder to collect instances into arrays. Standard Swift result builder semantics — if/else, for…in, and switch all work inside the body.

CLI — project init. The score CLI bootstraps new projects from one of the built-in example templates, giving you a working, runnable starting point instead of an empty scaffold:

score init                          // interactive picker
score init --template blog          // simple blog
score init --template marketing     // marketing site
score init --template docs          // documentation site
score init --template saas          // SaaS template (auth + CRUD + billing placeholder)
score init --template social        // social network (local-first + sync)
score init --template static        // no JS, no data — pure HTML/CSS
score init --template calculator    // multi-state derived UI, zero backend
score init --template minimal       // 1 page, 1 component, 1 API controller — blank canvas
score init --template localised     // all five locales, locale picker, localised formatting

Every template is the canonical example application for that use-case. They are not stubs — they are full, running Score apps you can read, modify, and deploy immediately. The minimal template is the exception: it is intentionally a clean slate — one page, one component, one controller — for developers who want to build from scratch without inheriting example opinions.

Theme

Score ships a complete default theme. You don't need to declare a theme at all — omit it entirely and Score's built-in defaults apply. When you do declare one, you override only what changes; everything else inherits from the defaults.

@Theme macro scope: When declared as a var theme property on Application, the @Theme macro is applied automatically by the protocol conformance — you never write the annotation yourself. Standalone theme declarations outside Application require the explicit @Theme annotation. Both forms produce identical output.
// No theme declaration needed. Score's built-in defaults apply.
@main
struct App: Application {
  var pages: [Page] { Home() }
}

// Or patch only what you need — everything else remains default.
@main
struct App: Application {
  var theme = Theme(
    roles: .override(
      accent: .blue(600),
      border: .slate(200)
    )
  )
  var pages: [Page] { Home() }
}

Built-in semantic roles (surface, text, border, accent, and more) are typed properties. Use them at call-sites for design-system correctness — a global accent change requires touching only the theme, not every component.

Custom roles are named extension points. You define the shades; Score generates a typed static func at build time so the call-site is identical to any built-in palette colour. Shade values can be palette colours (.neutral(500)) or explicit OKLCH for bespoke perceptual control:

@Theme
let theme = Theme(
  customRoles: [

    // Palette-scale shades — map to built-in colours.
    "info": .shades([
      50:  .sky(50),
      500: .sky(500),
      700: .sky(700)
    ]),

    // OKLCH shades — only lightness changes; chroma and hue are locked across the scale.
    "brand": .shades([
      50:  .oklch(0.97, 0.22, 265),   // near-white tint
      500: .oklch(0.55, 0.22, 265),   // primary
      600: .oklch(0.46, 0.22, 265),
      950: .oklch(0.12, 0.22, 265)    // near-black
    ])
  ],

  // Dark mode is declared inline — never a post-hoc mutation.
  // Only list the shades that need to shift for dark surface contrast;
  // all others inherit from the base declaration above.
  dark: ThemePatch(
    customRoles: [
      "brand": .shades([500: .oklch(0.68, 0.22, 265)])
    ]
  )
)

Use-sites are identical regardless of how a role was defined:

Text { "Heads up" }
  .background(.info(50))
  .border(width: 1, color: .info(200), style: .solid, radius: 10)

Button { "Get started" }
  .background(.brand(500))
  .font(color: .brand(50))

// The perceptual consistency of a locked chroma/hue scale means .brand(50) on a dark
// surface and .brand(950) on a light surface both read as "the same colour, different brightness".
Dark mode is first-class. Declare light and dark together at the @Theme site — mutating theme.dark afterwards is a compiler warning. The @Theme macro emits a typed static func for each custom role at the declaration site, immediately visible to SourceKit. .brand(500) is real Swift, not a string lookup. OKLCH literals used outside a theme declaration produce a compiler warning — add a custom role instead.

Theme is more than colour. It also controls typography, spacing scale, and border radius. All dimensions have sensible defaults — override only what you need.

@Theme
let theme = Theme(

  // ── Colour roles ────────────────────────────────────────────────────────
  roles: .override(accent: .blue(600)),
  customRoles: ["brand": .shades([…])],
  dark: ThemePatch(…),

  // ── Typography ──────────────────────────────────────────────────────────
  // Font families — defaults shown. Override any or all.
  fonts: .override(
    sans:  .systemUI,                          // default body font
    mono:  .custom("JetBrains Mono", fallback: .monospace),
    brand: .custom("Manrope", fallback: .sans) // used by .font(.brand, …)
  ),

  // Type scale — fluid steps, unitless ratio. Default is a minor third (1.25).
  typeScale: .modular(ratio: 1.25, base: 16),  // step 0 = 16px, step 1 = 20px, …

  // ── Spacing ─────────────────────────────────────────────────────────────
  // All .padding(…) and .margin(…) values reference the spacing scale.
  // Default unit is 4px. Override the multiplier to shift the whole scale.
  spacing: .scale(unit: 4),   // space-1=4px, space-2=8px, space-4=16px, …
                               // .scale(unit: 5) shifts everything up proportionally.

  // ── Radius ──────────────────────────────────────────────────────────────
  // Border radius tokens, referenced by integer step in .border(radius:).

Controllers and Routes

Backend API endpoints are grouped into Controller types that declare a base path and a set of Routes. Score defaults to JSON. Controllers are registered on the Application alongside pages.

struct PostController: Controller {
  var base: Path { "/api/posts" }

  var routes: [Route] {
    Route(.get,    "/",    use: index)    // GET  /api/posts
    Route(.post,   "/",    use: create)   // POST /api/posts
    Route(.patch,  "/:id", use: update)   // PATCH /api/posts/:id
    Route(.delete, "/:id", use: delete)   // DELETE /api/posts/:id
  }

  func index(req: Request) async throws -> [Post] {
    try await req.db.all(Post.self)
  }

  func create(req: Request) async throws -> Post {
    let body = try req.decode(Post.self, excluding: \.id)
    return try await req.db.insert(Post(id: .init(), title: body.title))
  }
}
No separate body types. Request decoding uses the Entity type directly. req.decode(T.self, excluding: \.id) accepts the entity shape minus the excluded fields (for creation). req.decode(T.self, partial: true) makes all fields optional (for PATCH updates). One type, one source of truth — no CreateBody or UpdateBody types required.

When a controller manages an Entity type, Score automatically generates _sync/pull and _sync/push endpoints under the same base path — no extra code required.

Middleware

Middleware conforms to the Middleware protocol and intercepts requests at the Application, Controller, or Route level. Middleware runs in declaration order — first declared, first executed.

// Define middleware by conforming to the protocol
struct AuthGuard: Middleware {
  func handle(req: Request, next: Next) async throws -> Response {
    guard let _ = try? req.auth.requireSession() else {
      throw AppError(.unauthorized, message: "Sign in required")
    }
    return try await next(req)
  }
}

struct RateLimit: Middleware {
  let max: Int
  let window: Duration

  func handle(req: Request, next: Next) async throws -> Response {
    let ip = req.remoteAddress
    let count = try await req.kv.increment("rate:\(ip)")
    guard count <= max else {
      throw AppError(.tooManyRequests, message: "Rate limit exceeded")
    }
    return try await next(req)
  }
}

// Apply at the controller level — all routes in this controller are guarded
struct TaskController: Controller {
  var base: Path { "/api/tasks" }
  var middleware: [Middleware] { AuthGuard(); RateLimit(max: 100, window: .minutes(1)) }

  var routes: [Route] { /* … */ }
}

// Apply at the application level — all routes are affected
@main
struct App: Application {
  var middleware: [Middleware] { RateLimit(max: 1000, window: .minutes(1)) }
  var pages: [Page] { /* … */ }
  var controllers: [Controller] { /* … */ }
}

// Apply to a single route
Route(.delete, "/:id", use: delete, middleware: [AuthGuard()])
Execution order: Application middleware runs first, then controller middleware, then route middleware. Within each level, middleware runs in declaration order. Every middleware must call next(req) to continue the chain or throw to short-circuit.

Error Handling

Controller handlers that throw produce structured error responses. Score catches thrown errors at the router level and serialises them based on the route's content type.

// Framework-provided protocol
protocol ScoreError: Error {
  var status: HTTPStatus { get }
  var message: String { get }
  var detail: String? { get }
}

// Framework-provided concrete type for common cases
struct AppError: ScoreError {
  let status: HTTPStatus
  let message: String
  var detail: String?
}

// Usage in a controller — throw to short-circuit
func update(req: Request) async throws -> Task {
  guard let task = try await req.db.find(Task.self, id: req.param("id")) else {
    throw AppError(.notFound, message: "Task not found")
  }
  return try await req.db.update(task, with: req.decode(Task.self, partial: true))
}

// Standard do/catch for granular recovery
func create(req: Request) async throws -> Task {
  let body = try req.decode(Task.self, excluding: \.id)
  do {
    return try await req.db.insert(Task(id: .init(), title: body.title))
  } catch let error as ScoreDBError where error.isConstraintViolation {
    throw AppError(.conflict, message: "Task already exists", detail: error.localizedDescription)
  }
}

Error responses: For JSON routes, thrown errors serialise as { "error": "Task not found", "status": 404 }. For HTML routes, Score renders the error page matching the status code if one is declared on the Application. Unhandled errors that don't conform to ScoreError produce a generic 500 response.

@main
struct App: Application {
  var errorPages: [ErrorPage] {
    NotFoundPage()       // matches 404
    ServerErrorPage()    // matches 5xx
  }
}

struct NotFoundPage: ErrorPage {
  static let status = HTTPStatus.notFound

  var body: some Node {
    Stack {
      Heading(.one) { "Page not found" }
      Text { "The page you're looking for doesn't exist." }
      Link(to: "/") { "Go home" }
    }
    .flex(.column, gap: 16, align: .center)
    .padding(48)
  }
}
Error hierarchy: ScoreError is the protocol — conform your own error types to it for custom status codes and messages. AppError is the framework-provided concrete type covering common cases. Both produce structured JSON or render error pages depending on the route's content type.

Components

A Component is a reusable node builder expressed as a struct. Components compose into pages and into other components. Props are plain stored properties; local reactive state uses @State.

struct UserCard: Component {
  let name:   String
  let avatar: String

  var body: some Node {
    Stack {
      Image(src: avatar, alt: name)
      Text { name }.font(.brand, size: 16, weight: .semibold)
    }
    .flex(.row, gap: 12, align: .center)
    .padding(inline: 16, block: 16)
    .border(width: 1, color: .border, style: .solid, radius: 8)
  }
}

data-scope is derived from the struct name (PascalCase → kebab-case). Within a scope, Score targets semantic HTML elements directly — no BEM classes, no manually assigned identifiers. When multiple elements of the same type exist in one scope, Score infers a data-key disambiguation from the action handler name, label text, or icon name — in that priority order. If two elements resolve to the same key, Score emits a compile error.

CSS is generated once per scope regardless of instance count — 100 UserCard renders produce 100 HTML nodes and one CSS block.

Scoping strategy. Score emits @scope ([data-scope="x"]) to ([data-scope]) for Baseline 2025+ browser targets. The lower limit to ([data-scope]) is the key invariant — styles hard-stop at any nested scope boundary, giving formal isolation guarantees without relying solely on compile-time data-key disambiguation. For older targets, Score falls back to :where([data-scope="x"]) which is zero-specificity and correct, just without the lower-limit boundary guarantee. Both forms are deduplicated — one CSS block regardless of instance count. The same browser targets configuration that governs the Temporal polyfill and anchor positioning fallback governs @scope vs :where emission.

Internationalisation and Localisation

Score uses Localizable.xcstrings — Apple's String Catalog format — as its single source of truth for all translations. The same file is used by both the Score web app and any companion SwiftUI app. There is no separate web i18n format. One file, one workflow, shared types.

Declaring locales. Supported locales are declared on the Application. Score ships with built-in translations for all ScoreUI copy (button labels, error messages, empty states) in the five declared locales. English is always the source language.

@main
struct App: Application {
  var locales = Locales(
    source:    .en,
    supported: [.en, .es, .it, .de, .ru],
    fallback:  .en           // shown when a key is missing in the active locale
  )
  var pages: [Page] { Home(); About() }
}

String Catalog file. Score reads Sources/App/Resources/Localizable.xcstrings — the same path Xcode uses. The format is identical to Apple's: JSON with a sourceLanguage key and a strings dictionary mapping keys to per-locale translations. Score's compiler reads this at build time and extracts all keys referenced in Swift source, warning on any that are missing translations in a declared locale.

Locale Detection and Switching

Score resolves the active locale in order: explicit cookie (as_locale) → Accept-Language header (SSR) → navigator.language (client) → declared fallback. The resolved locale is available as req.locale in controllers and as a reactive value in components.

// Server-side — read and set locale in a controller
func profile(req: Request) async throws -> ProfilePage {
  let locale = req.locale           // Locale — resolved from cookie or Accept-Language
  return ProfilePage(locale: locale)
}

// Manual override — sets as_locale cookie, redirects
func setLocale(req: Request) async throws -> Response {
  let tag = try req.decode(LocaleForm.self).tag  // e.g. "de"
  return try await req.locale.set(tag)           // sets cookie, returns redirect
}

// In a component — LocalePicker is a ScoreUI component
struct AppNav: Component {
  var body: some Node {
    Stack {
      /* … navigation … */
      LocalePicker()          // renders a <select> or button group; wires set-locale route
    }
  }
}

The LocalePicker ScoreUI component works without JS — it submits a form to a Score-generated POST /_score/locale endpoint that sets the cookie and redirects back. With JS, the switch is instant via the data runtime without a page reload.

Dates, Numbers, and Currencies

Score never formats dates or numbers manually. All locale-aware formatting goes through Swift's FormatStyle on the server and Intl in the browser runtime — both driven by the active locale.

// Server-side — Swift FormatStyle, locale-aware
Text { Date.now.formatted(.dateTime.day().month(.wide).year(), locale: req.locale) }
// → "27 February 2026" (en) / "27 de febrero de 2026" (es) / "27 февраля 2026 г." (ru)

Text { 12450.50.formatted(.currency(code: "EUR"), locale: req.locale) }
// → "€12,450.50" (en) / "12.450,50 €" (de) / "12 450,50 €" (ru)

// Browser-side — Intl, driven by locale cookie
RelativeTime(post.createdAt)     // → "2 hours ago" / "hace 2 horas" / "2 часа назад"
Date formatting is never hardcoded. Score emits a lint error for any date formatted with string concatenation or a literal format string (e.g. "dd/MM/yyyy"). All dates must go through FormatStyle (server) or Intl.DateTimeFormat (browser). The locale determines the calendar, separator, and field order — the developer never does.
RTL and Writing Direction

When the active locale uses a right-to-left script (Arabic, Hebrew, Persian), Score sets dir="rtl" on the <html> element. Because all Score-generated CSS uses logical properties (padding-inline, margin-block, border-inline-end, inset-inline-start), the layout flips automatically — no RTL override stylesheet, no duplicated rules. The five launch locales (en, es, it, de, ru) are all LTR; this is groundwork for future RTL locale support.

  const getScope = (el)      => closest(el, "[data-scope]");

  const stateOwner = (scope, name) =>
    scope?.querySelector(`[data-as-value\\:${CSS.escape(name)}]`) ?? null;

  const readAllState = (scope) => {
    const owner = scope?.querySelector("[data-as-value]") ?? null;
    if (!owner) return {};
    return Object.entries(owner.dataset)
      .filter(([k]) => k.startsWith("asValue:"))
      .reduce((acc, [k, v]) => {
        const key = k.slice(8); // strip "asValue:"
        const n = Number(v);
        acc[key] = Number.isFinite(n) && v.trim() !== "" ? n : v;
        return acc;
      }, {});
  };

  const writeState = (scope, name, next) => {
    const owner = stateOwner(scope, name);
    if (!owner) return;
    const str = String(next);
    owner.setAttribute(`data-as-value:${name}`, str);
    scope.querySelectorAll(`[data-as-bind="${name}"]`)
         .forEach(el => { el.textContent = str; });
    scope.querySelectorAll(`[data-as-variant\\:${CSS.escape(name)}]`)
         .forEach(el => {
           el.setAttribute(`data-as-variant:${name}`, str);
           if (str === "true")  el.removeAttribute("hidden");
           if (str === "false") el.setAttribute("hidden", "");
         });
    const ctrl = scope.querySelector(`[aria-controls][data-as-action]`);
    if (ctrl && name === "expanded") ctrl.setAttribute("aria-expanded", str);
  };

  // Action stubs compiled from @Action bodies — appended per page by as-actions.js
  const actions = window.__asActions__ ?? {};

  document.addEventListener("click", e => {
    const trigger = closest(e.target, "[data-as-action]");
    if (!trigger) return;
    const scope      = getScope(trigger);
    const scopeName  = scope?.dataset.scope;
    const actionName = trigger.dataset.asAction;
    const arg        = trigger.dataset.asActionArg;
    const stub       = actions[scopeName]?.[actionName];
    if (!stub) return;
    const snapshot = readAllState(scope);
    const next = stub({ ...snapshot }, arg);
    Object.entries(next).forEach(([key, val]) => writeState(scope, key, val));
  });

  // Input capture — direct value binding, no action needed
  document.addEventListener("change", e => {
    const input = closest(e.target, "[data-as-target]");
    if (!input) return;
    writeState(getScope(input), input.dataset.asTarget, input.value);
  });
})();

Compilation model: Handler closures are never shipped to the browser. Score analyses them at build time and emits data-as-action attributes referencing named compiled stubs. The runtime is a dispatcher — it calls a named stub with a state snapshot and writes the returned changes. No string evaluation, no Function(), no arbitrary code execution. The Swift source is never shipped; the compiled intent is.
The boundary: @Action handles all client-side state mutations — simple or complex, single or multi-target. The moment a mutation needs async, a network call, or a database write — use a form submission to a controller route. These are not workarounds; they are the correct boundaries.
Three handler types — know the difference:
1. @Action functions: Pure state mutations. Compiled to named stubs. No side effects. No async. No network. No DOM.
2. Inline closures calling @Action: .click { state in state.increment() } — delegates to the stub. Same constraints.
3. Inline closures constructing Form: .click { Form(action: …, method: .patch) { … } } — compiles to DOM form construction + submission. This is the prescribed path for network mutations from an event handler. It is not an @Action violation because it is not an @Action.
Form contract: A Form constructed inside a .click or .change handler auto-submits — no .submit() modifier required. For standalone Form nodes in the body tree, the compiler warns if no Button(.submit) or Input(.submit) child exists. The default form action is a POST to the declared route; override by pointing the action: parameter to a different route.
@Action — named compiled mutations

All client-side state mutations go through named @Action functions on the component. Actions are analysed at build time and compiled to named stubs in as-interactivity.js. They run client-side, are fully type-checked at build time, and enforce hard constraints: no async, no network, no DOM access, no side effects. Actions mutate state directly via inout assignment — Score analyses what was assigned and emits only those keys to the runtime dispatcher.

struct ShoppingCart: Component {
  @State var items:    [CartItem] = []
  @State var subtotal: Double     = 0
  @State var discount: Double     = 0
  @State var total:    Double     = 0

  @Action func addItem(_ item: CartItem) {
    items.append(item)
    subtotal = items.reduce(0) { $0 + $1.price }
    discount = subtotal > 100 ? subtotal * 0.1 : 0
    total    = subtotal - discount
  }

  @Action func removeItem(id: UUID) {
    items    = items.filter { $0.id != id }
    subtotal = items.reduce(0) { $0 + $1.price }
    total    = subtotal - discount
  }

  var body: some Node {
    Stack {
      for item in items {
        CartRow(item: item)
          .click { state in state.removeItem(id: item.id) }
      }
      Text { "Subtotal: \(subtotal)" }
      Text { "Discount: \(discount)" }
      Text { "Total: \(total)" }.font(.brand, size: 20, weight: .semibold)
    }
    .flex(.column, gap: 12)
  }
}

The body compiles to a named stub in as-interactivity.js that receives a JSON state snapshot, applies the mutation, and returns only the changed keys. The Swift source is never shipped.

Built-in atomic operations (no @Action needed):

For the simplest cases — a single field, no branching, no cross-state reads — Score recognises a set of direct inline assignments that compile to canned atomic ops without requiring a named @Action:

// These compile directly to atomic ops — no @Action declaration needed
.click { state in state.count += 1 }          // increment
.click { state in state.count -= 1 }          // decrement
.click { state in state.isOpen.toggle() }     // toggle
.click { state in state.status = "Saved!" }   // set

// Input binding — .bind($property) wires the input value to @State directly
Input(type: .text, placeholder: "Search…")
  .bind($query)                                // two-way binding, no handler needed
Rule: If the mutation touches more than one state field, branches on current state, or derives a value — declare a named @Action. The inline form exists only for truly trivial single-field mutations.
@Binding — two-way state references

@Binding creates a two-way reference to a parent component's @State property. The parent passes $count (projected value); the child receives it as @Binding var count: Int. Mutations to the binding propagate to the parent's state. Use for decomposing complex components that share state.

// Parent passes its @State as a binding
struct ParentComponent: Component {
  @State var count: Int = 0

  var body: some Node {
    Stack {
      Text { "\(count)" }
      StepperControl(count: $count)  // pass binding
    }
  }
}

// Child receives the binding — mutations flow back to parent
struct StepperControl: Component {
  @Binding var count: Int

  @Action func increment() { count += 1 }
  @Action func decrement() { count -= 1 }

  var body: some Node {
    Stack {
      Button { "−" }.click { state in state.decrement() }
      Text { "\(count)" }
      Button { "+" }.click { state in state.increment() }
    }
    .flex(.row, gap: 8, align: .center)
  }
}
Note: @Binding for component-to-component state sharing is distinct from .bind($property) on input nodes. .bind() wires an HTML input element's value to a @State property. @Binding wires a child component's property to a parent component's @State.

Multi-state example — TipCalculator

Three named states, one cross-state @Action. The calculate action reads both bill and tipPercent to derive label — Score detects this from the action body and emits the correct attribute wiring automatically.

struct TipCalculator: Component {
  @State var bill:       Double = 0
  @State var tipPercent: Double = 15
  @State var label:      String = "Enter a bill amount"

  @Action func adjustTip(by delta: Double) {
    tipPercent = max(0, min(100, tipPercent + delta))
  }

  @Action func calculate() {
    let tip   = bill * (tipPercent / 100)
    let total = bill + tip
    label = "Tip: $\(tip.formatted()) · Total: $\(total.formatted())"
  }

  var body: some Node {
    Stack {
      Input(type: .number, placeholder: "Bill amount")
        .bind($bill)

      Stack {
        Button { "−" }
          .click { state in state.adjustTip(by: -5) }
        Text { "\(tipPercent)%" }
        Button { "+" }
          .click { state in state.adjustTip(by: 5) }
      }
      .flex(.row, gap: 8, align: .center)

      Button { "Calculate" }
        .click { state in state.calculate() }

      Text { label }.font(.brand, size: 16, weight: .semibold)
    }
    .flex(.column, gap: 12)
    .padding(24)
  }
}

Variants

Variants express conditional styling driven by state. They compile to data-* attributes in HTML; the generated CSS targets those attributes directly. No inline styles, no class toggling.

The variant key is derived automatically from the bound state property — $isOpen becomes data-as-variant:isOpen. The name: parameter is only needed when two variants on the same element reference different state keys and require CSS disambiguation.

// name: derived — Score infers it from $isOpen
Button { "Menu" }
  .click { state in state.toggleOpen() }
  .variant($isOpen) { open in
    open
      ? $0.background(.neutral(900)).font(color: .neutral(0))
      : $0.background(.neutral(100)).font(color: .neutral(900))
  }

// name: explicit — only when two variants on the same element need disambiguation
Button { … }
  .variant($isOpen, name: "open") { … }
  .variant($isDisabled, name: "disabled") { … }

Score emits data-as-variant:isOpen="true|false" on the element; generated CSS targets those attribute values scoped to the component.

Animations

Score's animation model is declarative: you express intent, not implementation. The compiler emits CSS that targets only GPU-compositable properties — transform, opacity, and filter. Properties that trigger layout (height, width, margin, top) are not animatable through Score modifiers. This is a constraint, not a limitation — it is the reason animations are always smooth.

Transitions express how a style change moves from one value to another. Apply them to any node; they are automatically scoped and emitted as CSS transition declarations:

Button { "Save" }
  .transition(.opacity, duration: 0.15, easing: .easeOut)
  .transition(.transform, duration: 0.2, easing: .spring(damping: 0.7))

// Multiple properties in one call
Stack { … }
  .transition([.opacity, .transform], duration: 0.25)

Keyframe animations are defined inline or as named sequences. Named keyframes are deduplicated at build time — 100 components sharing the same keyframe definition emit one @keyframes block:

// Inline — anonymous, single-use
Image(src: logo, alt: "Logo")
  .animate(
    keyframes: [
      .from(.opacity(0), .transform(.translateY(-8))),
      .to(.opacity(1),   .transform(.translateY(0)))
    ],
    duration: 0.3,
    easing:   .easeOut,
    fill:     .forwards
  )

// Named — reusable, deduplicated, uses the same modifier API as nodes
@Keyframes("fade-up")
let fadeUp = Keyframes {
  Frame(at: .from)
    .opacity(0)
    .transform(.translateY(-8))
  Frame(at: .to)
    .opacity(1)
    .transform(.translateY(0))
}

Text { "Welcome" }
  .animate(.named("fade-up"), duration: 0.3, delay: 0.1)

Scroll-driven animations link an animation's progress to scroll position using the CSS Scroll Timeline API — no JS required:

// Reveal on scroll — plays as the element enters the viewport
Stack { … }
  .scrollReveal(.fadeUp, threshold: 0.2)

// Progress bar that fills as the user scrolls the page
ProgressBar()
  .scrollProgress(.horizontal, source: .page)

ScoreUI components animate by default. Every interactive component in the library — Dialog, Dropdown, Sheet, Toast, Accordion, Tooltip — ships with entrance/exit animations using only opacity and transform. They require no configuration and are suppressible per-instance:

Dialog { … }              // animated by default
Dialog { … }.animated(false)  // suppress for this instance
Prefers-reduced-motion is automatic. Every Score-generated animation — modifiers, named keyframes, scroll reveals, and ScoreUI defaults — is wrapped in a @media (prefers-reduced-motion: no-preference) block. The developer never writes this media query. When the user has enabled reduced motion, the animation declarations are not applied — no transitions, no keyframes, no scroll reveals. Static variants remain fully functional. This is a framework invariant, not a toggle.
GPU compositing guarantee: Score's type system prevents animating layout-triggering properties. Passing .height, .width, .margin, or .top to .transition() or .animate() is a compile error. If a layout change is unavoidable, use transform: scaleY() as the composited equivalent — Score provides .transform(.scaleY(0)) for exactly this case.

The runtime activation matrix gains one row: .scrollReveal() and .scrollProgress() emit a small as-scroll.js module only when used. Pure CSS animations (.transition(), .animate(), named keyframes) require no JS.

Data (Local-first + Sync)

When data features are used, Score is local-first by default. Pages read from a local store first, mutations apply optimistically, and a background sync engine reconciles with the backend. Developers write a normal REST mental model; replication is framework-managed.

Entity

Define only domain fields. Score stores sync metadata in a parallel table alongside the entity — never inside it. Your type stays clean; the framework's bookkeeping is invisible to user code.

Temporal and Dates

Score uses Swift's Date and Calendar for all server-side date logic — they are correct, well-understood, and available without any JS. For anything that reaches the browser, Score uses the Temporal API exclusively. new Date() does not appear anywhere in Score's generated JS.

The boundary is the data-attribute encoding. When a @State value holds a date, Score encodes it as an ISO-8601 string in the data-as-value:* attribute. The runtime reads and writes that string via Temporal.Instant.from(). Timezone-aware display uses Temporal.Now.zonedDateTimeISO() — the user's local zone is respected automatically.

Transaction semantics

Score's sync model makes explicit guarantees that must be locked in early — sync systems calcify quickly and retrofitting consistency guarantees is expensive.

GuaranteeRuling
Single-entity writes Atomic. A PATCH /api/todos/:id either fully applies or fully fails. Partial application is not possible.
Multi-entity writes Not atomic by default. Each entity mutation is an independent operation in the outbox. If you need atomicity across entities, wrap them in an explicit Transaction block (see below).
Optimistic application Applied immediately to the local store. Rolled back if the server rejects. Rollback is surfaced as a Conflict with reason: "rejected".
Outbox ordering Mutations within the same entity are applied in outbox order (FIFO). Mutations across different entities have no ordering guarantee.
Idempotency Every sync push carries an idempotency key. Duplicate pushes (e.g. after a network retry) are detected server-side via ScoreKV and ignored.
Consistency level Eventual. The local store converges to the server state after sync. There is no linearisability guarantee across clients.
Read-your-writes Guaranteed locally. After an optimistic write, reads on the same client see the written value immediately. Other clients see it after the next successful pull. 
// Explicit transaction — multi-entity atomic write.
// Score wraps both mutations in a single SQLite transaction server-side.
// Both succeed or both fail; the outbox treats them as one unit.
try await req.db.transaction {
  let order = try await $0.insert(Order(id: .init(), userId: user.id, total: cart.total))
  try await $0.insert(OrderLine(orderId: order.id, items: cart.items))
}

// Without a transaction block, these are two independent outbox entries:
try await req.db.insert(order)   // could succeed
try await req.db.insert(line)    // could fail independently
Rule: If you need atomicity across entities, declare it with a transaction { } block. Score never infers transaction boundaries.

Runtime Activation Matrix

Runtime activation is decided purely from AST analysis at build time — no heuristics, no runtime introspection. The rules below are exhaustive. If a case is not listed here, the answer is: no runtime emitted.

Implementation mechanism: Score's build-time analysis is implemented as a Swift Macro package using SwiftSyntax. @Action, @State, @Theme, @Query, and @Keyframes are all macro annotations that expand at compile time. The score build CLI invokes swift build with the Score macro plugin — no separate compiler pass or external build tool.
Rule: A runtime artefact is emitted if and only if a triggering feature is statically present in the AST. #if blocks are evaluated at build time; conditional runtime logic is not a trigger.
Feature usedArtefact emittedEdge condition / ruling
@State declared, mutated as-interactivity.js Normal case.
@State declared, never mutated None Score emits a compiler warning: "@State 'x' is never mutated — use a let property instead." No JS emitted.
@Action defined as-interactivity.js + named stub Always — even if the action is never called. Unused actions produce a compiler warning. Stubs are the only mutation path; no string evaluator is emitted.
Event modifier (.click, .change, etc.) as-interactivity.js Always, regardless of whether the handler writes any state.
@Query — dynamic predicate as-data.js + backend binary Normal case. Query results update reactively.
@Query — static predicate (all values known at build time) None (inlined at build time) Score evaluates the query at build, inlines results as static HTML. No data runtime needed. Recompile to refresh.
@Query declared, page not rendered at runtime as-data.js + backend binary Still emitted. The query declaration is sufficient. Score does not track whether a page is reachable.
Controller registered Backend binary Always. A registered controller implies a running server even if no pages use its data.
Auth features used (req.auth, session reads) Backend binary Always. Auth requires a running server for session validation.
Conditional route (if featureFlag { Route(…) }) Backend binary (if any branch has a controller) Both branches are analysed. If either branch would trigger backend emission, backend is emitted. Feature flags do not suppress runtime.
@State or event modifier inside a component used only on static pages as-interactivity.js on those pages only Emission is per-page. A component with @State used on page A does not trigger JS emission on page B.

Every surprising case in the table produces a compiler warning or note — "why did runtime turn on?" should never be a mystery.

PWA Support and Cross-Platform Strategy

Engineering effort is concentrated on the Apple/SwiftUI path — the PWA path provides broad compatibility without compromise.

Content Security Policy

Score generates all JS as static files with deterministic content. In static build mode, Score computes the SHA-256 hash of each emitted JS file and generates a Content-Security-Policy header fragment with the corresponding script-src 'sha256-…' values — no nonces required because the content never changes. In runtime mode, Score generates a per-request nonce and injects it into the CSP header and all inline script tags automatically. The developer declares the base policy; Score fills in the script directive:

@main
struct App: Application {
  var csp = ContentSecurityPolicy(
    defaultSrc: ["'self'"],
    imgSrc:     ["'self'", "data:", "https://cdn.example.com"],
    connectSrc: ["'self'", "https://api.example.com"]
    // script-src is managed by Score — do not override it
  )
}

2a. CSS Emission Standards

Score's CSS compiler follows a strict set of modern CSS rules. These are not preferences — they are invariants enforced at build time. The compiler will warn or error on violations.

Logical Properties — Required

All Score-generated CSS uses logical properties for anything directional. Physical properties (margin-left, padding-right, border-top, top/left/bottom/right in positioned layouts) are forbidden in generated output. This is a build-time error, not a warning — it is the foundation of automatic RTL support.

Physical (never emitted)Logical (always emitted)
margin-left / margin-rightmargin-inline-start / margin-inline-end
padding-top / padding-bottompadding-block-start / padding-block-end
border-top / border-bottomborder-block-start / border-block-end
left / right (in inset)inset-inline-start / inset-inline-end
border-top-left-radiusborder-start-start-radius
width / height (as constraints)inline-size / block-size
min-width / max-heightmin-inline-size / max-block-size
text-align: lefttext-align: start
float: leftfloat: inline-start

Four-sided shorthands where all four sides are equal (padding: var(--as-space-4), margin: 0, inset: 0) are acceptable — they are symmetric and have no directional intent. Two-value shorthands that mix block and inline (padding: 8px 16px) are expanded to their logical longhand equivalents at compile time.

Container Queries — Component-Level Breakpoints

Score supports two breakpoint systems. Viewport-based breakpoints (.compact { }, .tablet { }, etc.) are the existing system — they emit @media (min-width: …) rules. Container-based breakpoints are a second system for components that need to respond to their parent container's size rather than the viewport:

// Container query — responds to the parent container's inline size
// Score emits: container-type: inline-size on the component's root
// and @container (min-inline-size: …) rules for the breakpoints
Card { … }
  .container { $0.flex(.row, gap: 16) }     // Card switches to row when container ≥ 320px
  .container(min: 480) { $0.padding(24) }   // explicit threshold

// Viewport breakpoints remain available for layout-level decisions
Stack { … }
  .tablet { $0.grid(columns: 2, gap: 24) }  // 2-column grid at tablet viewport width
Rule: Use container queries for component-level adaptation (how a Card looks inside a sidebar vs a main column). Use viewport breakpoints for layout-level decisions (how many columns the page grid has). Score does not enforce this distinction — it is a design judgment — but the generated output for each is unambiguous.

Fluid Typography with clamp()

Score's type scale uses clamp(min, preferred, max) for fluid sizing rather than discrete step values at each breakpoint. The preferred value uses container query units (cqi) so type scales to its container, not the viewport. This means a heading inside a narrow sidebar column is appropriately smaller than the same heading in a full-width section — with no breakpoint modifier required.

/* Score emits fluid type scale — example for --as-font-step-2 */
--as-font-step-2: clamp(1.25rem, 1rem + 2cqi, 1.75rem);
--as-font-step-3: clamp(1.5rem,  1rem + 3cqi, 2.25rem);
--as-font-step-4: clamp(2rem,    1rem + 5cqi, 3rem);

Modern Selectors — :has()

Score uses :has() for parent-aware styling that would otherwise require JS class toggling. In generated CSS, :has() is used to style form groups based on their input's validity state, to style list containers based on whether they have children, and in ScoreUI's Checkbox, Switch, and RadioGroup components to apply checked-state styles to the label without JS:

/* ScoreUI Checkbox — label style driven by :has(), no JS class toggle */
@scope ([data-scope="checkbox"]) to ([data-scope]) {
  :scope:has(input:checked) [data-key="label"] {
    color: var(--as-color-accent);
    font-weight: var(--as-weight-semibold);
  }
  :scope:has(input:invalid) [data-key="error"] {
    display: block;
  }
}

Performance — content-visibility

Score emits content-visibility: auto on page sections that are below the fold at initial render. This instructs the browser to skip layout and paint for off-screen content until it scrolls into view — a significant rendering performance win on content-heavy pages at no JS cost. Score detects this automatically from component position in the page tree and does not require developer annotation.

Text Rendering

Score's base CSS layer sets text-wrap: balance on all headings (h1–h4) and text-wrap: pretty on body text. Both are Baseline 2024. balance prevents awkward single-word orphans in headings; pretty avoids orphans in paragraph text. Neither property affects layout — they are pure typographic quality improvements at zero cost.

Colour — color-mix()

Derived colours in Score's theme system use color-mix(in oklch, …) rather than hardcoded intermediate values. This means tints, shades, and opacity variants are computed relative to the theme token at runtime — changing a single brand colour cascades correctly through all derived values without regenerating the token sheet.

/* Score emits derived colours via color-mix — not hardcoded intermediate values */
--as-color-accent-subtle:  color-mix(in oklch, var(--as-color-accent) 15%, transparent);
--as-color-accent-hover:   color-mix(in oklch, var(--as-color-accent) 85%, white);
--as-color-accent-border:  color-mix(in oklch, var(--as-color-accent) 40%, transparent);

Scrollbar Policy

Score does not hide scrollbars by default. Native scrollbar behaviour is preserved as-is. When a scrollbar-free presentation is intentional — e.g. a horizontal scroll carousel or a custom overflow panel — apply the .scrollbar(.hidden) modifier to that specific container. Scrolling behaviour is always preserved; only the visual track is affected.

3. Rendering Model

Score supports two build modes, detected automatically:

  • Static build: routes compile to decoupled files (.html + as-*.css + optional as-*.js). Serve from any host — CDN, S3, Stage, Nginx.
  • Runtime build: routes compile into the server binary for request-time rendering. Generated assets remain static files. The output stays inspectable.

Runtime activation is determined at build time across three independent triggers:

DimensionTriggerResult
JS / InteractivityAny event handler or @State on that pageJS bundle emitted for that page only
JS / DataAny @Query or @Store on that pageData runtime emitted for that page only
BackendAny DB access, auth, or server-side side effect on that routeSite compiled as a runtime binary

4. Progressive Rendering Model

Each layer is independently useful. HTML works without CSS. CSS works without JS. JS adds optional interactivity without a backend. Runtime enables personalisation and durable side effects. No layer requires the one above it — this is a correctness model, not a performance strategy. Interactivity and runtime are orthogonal: a page can be interactive without a server, and a server can run without shipping any JS.

5. API Model

JSON over REST for all API routes at launch. Swift types define schemas; serialisation is generated automatically. REST semantics are preserved with idempotency respected. All JSON is browser-native — no JS decoding step, no compromise to the no-JS baseline.

Future: Protocol Buffer support as an opt-in layer for teams needing binary efficiency at scale. Explicitly not a launch requirement and will never be the default wire format — unless Protocol Buffers achieve native browser baseline support, at which point the default will be reconsidered.

7. Durability (Runtime Mode)

When the backend runtime is active, Score enforces durability primitives automatically: timeout enforcement, bounded retries with exponential backoff and jitter, idempotency keys with replay protection, circuit breakers, rate limiting, structured errors, and graceful degradation. These are on by default and cannot be disabled.

8. Dependency Policy

Allowed without approval: the four curated Swift Package Index collections (Apple, Swift.org, SSWG, Swift Server Community). Any dependency outside these requires explicit justification. Commonly useful packages from these collections include swift-http-types, swift-collections, swift-log, swift-crypto, and swift-markdown. On the JS side, @js-temporal/polyfill is the only permitted client-side runtime dependency — included automatically by Score when targeting browsers without native Temporal support. Listing a package is not approval — and before writing any library-level code, search these collections first. The solutions for cryptography, HTTP, concurrency, and logging already exist there.

9. Non-Goals

  • Hydration-heavy SPA architecture
  • Global JS runtime
  • Mandatory client frameworks
  • Animating layout-triggering CSS properties — only GPU-compositable properties (transform, opacity, filter) are animatable through Score modifiers
  • Emitting physical CSS directional properties (margin-left, padding-right, border-top) in generated output — all directional values use logical properties
  • Pluggable database drivers at launch — additional drivers require an RFC

📙 Allegro Stage — Product Requirements

Swift-native hosting • framework-agnostic • multi-mode operation

1. Overview

Allegro Stage is a Swift-native hosting and deployment platform for static outputs, runtime binaries, Score apps, generic Swift packages, and standalone Swift executables. Stage remains framework-agnostic.

2. Product Goals

Swift-native hosting
Swift workloads are first-class.
Multi-mode operation
Local dev • managed/serverless • self-hosted.
Universal Swift workloads
Executables • packages • precompiled binaries.
Score supported, not required
Stage is not a Score dependency.

Score builds applications. Stage hosts applications.

  • Stage must not depend on Score internals
  • Stage must not be implemented as a Score application
Stage marketing/docs/dashboard UI may use Score. Core infrastructure orchestration must remain Score-independent.

  • stage dev

  • stage deploy
  • stage serve
  • stage logs

  • Locking users into Score
  • Reimplementing mature Swift server tooling unnecessarily

📜 RFC-0001 — Allegro Architecture Overview

Status: Active • Normative RFC

RFC • boundaries

1. Purpose

Define architectural boundaries and invariants for Allegro Ecosystem, Score, and Stage.

2. System Overview

Three layers: Score (framework), Stage (hosting), Libretto and user apps (application). Each layer has a defined boundary — no layer may depend on the internals of a layer it does not own.

3. Architectural Invariants

Static-first invariant: if runtime features are not used, output must be static.
Automatic runtime invariant: runtime activates via feature detection.
Wire format invariant: all API routes transmit JSON at launch. Protobuf is a future opt-in only — the default must not compromise the no-JS baseline.

4. Persistence Model

ScoreDB is a behaviour-driven interface: the API is stable regardless of the underlying storage engine. At launch: SQLite for durable persistence + custom SwiftNIO-based in-memory store. Alternative drivers (e.g. PostgreSQL) may be added via RFC without changing the consumer API.

5. Durability Model

Runtime mode enforces timeouts, bounded retries, idempotency protection, circuit breakers, rate limiting, and structured errors — preventing duplicate side effects, cascading failures, and infinite retry loops.

6. Hosting Boundary

Score compiles applications and produces static and runtime artefacts. Stage hosts workloads and manages deployment, scaling, logs, and environments. Stage marketing and dashboard UI may use Score; core infrastructure must remain Score-independent.

7. Dependency Governance

Dependencies must originate from curated Swift Package Index collections; any external dependency requires explicit RFC and justification.

Deviation policy: Allegro enforces recommendations but permits informed deviations. The supported path is documented. If you deviate — custom wire format, non-approved dependency, alternative storage — the support level drops and you are explicitly in unsupported territory. This mirrors how Swift handles unsafe APIs: the escape hatch exists, but you know you have left the safe zone.
