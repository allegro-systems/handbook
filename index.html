Design Philosophy

principles + style guide

Product Standards

  • Interfaces should feel immediate. Most interactions should acknowledge input within a fraction of a second.
  • Navigation should stay shallow and legible.
  • Copy should be direct, active, and short.
  • Motion should clarify state changes, not decorate them.
  • Hit targets must stay comfortable on touch devices.
  • Empty, loading, and error states should look intentional.
  • Visual hierarchy should come from spacing, typography, and contrast before ornament.
  • Accessibility is a baseline quality requirement.

What We Avoid

  • Autoplay and attention traps
  • First-load modal interruptions
  • Excessive tooltip dependency
  • Deeply nested navigation for simple products
  • Full-page loading spinners as the default answer to latency
  • Decorative motion that obscures information

Ecosystem PRD

systems thesis · boundaries · dependencies · progress

Overview

Allegro is a Swift-first web ecosystem made of five products:

  • Score — the framework and rendering/runtime foundation
  • Stage — the hosting and operations product
  • Libretto — the first-party reference application
  • Allegro Site — the canonical public site and documentation surface
  • Composer — native macOS and iPad visual editor

Systems Thesis

Most pages should be static. Runtime should appear only when the problem requires it. Performance and durability are product qualities, not optional polish.

  • Static is the base state. Dynamic behaviour must justify its cost.
  • Runtime is a capability. Not a mode switch to micromanage.
  • Progressive enhancement is required. HTML → CSS → JS/server.
  • Performance is correctness. Slow output is an architecture failure.
  • Durability matters when state matters. Retries, timeouts, idempotency.
  • Boundaries stay hard. Score builds. Stage hosts. Libretto proves.
  • Dependencies are architecture. Curated, not procured.
  • Narrow focus is intentional. One coherent path for typed, static-first web.

Delivery Order

  1. Score becomes viable enough to build and ship real applications.
  2. Stage becomes viable enough to host those applications well.
  3. Libretto uses both in production and exposes the rough edges.
  4. Allegro Site publishes ecosystem documentation.
  5. Composer provides visual editing on top of the canonical CLI toolchain.

Architecture Decision Records

ADR-001 · FoundationDB KV Store

Single transactional KV abstraction. Minimal API: get, set, delete, scan, transactions. No SQL in core.

ADR-002 · TC39 Signals Polyfill

Browser runtime targets signal-polyfill. @State → Signal.State, @Computed → Signal.Computed. Three scopes: Application, Page, Element.

ADR-003 · Single-Worker Composer

One sandboxed worker at a time. Inactive projects compressed and evicted.

ADR-004 · Swift 6 Strict Concurrency

All Node types must be Sendable. Structured concurrency throughout.

ADR-005 · Static-First Default

Static output is default. Runtime capabilities opt-in. Capability manifest enforces at build/deploy.

ADR-006 · Deterministic Builds

Identical inputs → byte-equivalent outputs.

ADR-007 · Three-Level Signal State

Application (global singletons), Page (module-scoped, torn down on nav), Element (instance-scoped, per mount).

Score · Swift LikeButton.swift 
// Element — stateful, instance-scoped
// Each mount gets isolated signals

struct LikeButton: Element {
    @State var liked: Bool = false
    @State var count: Int = 0

    @Computed
    var label: String {
        liked ? "♥ \(count)" : "♡ \(count)"
    }

    @Action
    func toggle() {
        liked.toggle()
        count += liked ? 1 : -1
    }

    var body: some Node {
        Button { label }
            .on(.click, action: "toggle")
    }
}
Compiled Output like-button.js 
import { Signal } from "signal-polyfill";

// Element: factory — per mount
export function mountLikeButton(el) {
  // @State — instance-scoped
  const liked =
    new Signal.State(false);
  const count =
    new Signal.State(0);

  // @Computed — derived, memoised
  const label =
    new Signal.Computed(() =>
      liked.get()
        ? `♥ ${count.get()}`
        : `♡ ${count.get()}`
    );

  // @Action
  const toggle = () => {
    liked.set(!liked.get());
    count.set(count.get()
      + (liked.get() ? 1 : -1));
  };

  el.addEventListener("click",
    toggle);

  return { liked, count, label,
           toggle };
}
Lifetime: Element mount to unmount. Each instance is fully isolated. Use for interactive controls — counters, pickers, toggles, forms.

// Score runtime — quantity-picker.js
// Auto-generated. Do not edit.
import { Signal } from "signal-polyfill";

// Element: factory — called per mount
export function mountQuantityPicker(
  el, { product, max = 10 }
) {
  // Element: @State — instance-scoped
  const count =
    new Signal.State(0);

  // @Computed
  const canIncrement =
    new Signal.Computed(
      () => count.get() < max
    );

  // @Action
  const increment = () => {
    if (canIncrement.get())
      count.set(count.get() + 1);
  };
  const decrement = () => {
    if (count.get() > 0)
      count.set(count.get() - 1);
  };

  // Wire DOM + return teardown
  return { count, canIncrement,
           increment, decrement };
}
Lifetime: Element mount to unmount. Each instance is fully isolated.

Lowering Summary

Swift construct JS output Scope
@State on Application export const x = new Signal.State(v) in app.js Module singleton (global)
@State on Page const x = new Signal.State(v) in page module Module-scoped (page lifetime)
@State on Element const x = new Signal.State(v) inside factory Instance-scoped (per mount)
@Computed new Signal.Computed(() => ...) Same scope as containing type
@Action Plain function performing .set() calls Same scope as containing type

Reactive Invariants

  • Fine-grained dependency tracking. Dynamic dependency graph based on actual reads.
  • Batched updates. Multiple writes in same microtask coalesce. No cascading DOM mutations.
  • Deterministic effect ordering. Parent before children, earlier before later.
  • Glitch-free derived state. Computed values never see partial updates.
  • Disposal semantics. Removed subtrees dispose all bindings. No subscription leaks.
  • Scope isolation. Application is global, Page is module-scoped, Element is instance-scoped.

Storage

One transactional KV store with hierarchical keys. Store protocol with async methods; InMemoryStore actor for development, FoundationDB for production.

Storage CRUD & Codable ScoreStorage · Key + Store protocol
Score · Swift UserService.swift 
// Hierarchical keys — variadic segments
let key = Key("users", userId, "profile")
// key.description → "users.42.profile"

// Codable convenience — type-safe
try await store.set(user, forKey: key)
let user = try await store.get(
    User.self, forKey: key
)

// TTL — auto-expiry for sessions
try await store.set(
    token, forKey: sessionKey,
    ttl: .seconds(3600)
)

// Check existence without reading
let exists = try await store.exists(key)

// Delete
try await store.delete(key)
Score · Swift Analytics.swift 
// Prefix scan — ordered async stream
for try await (key, data) in
    store.scan(prefix: Key("users"))
{
    let user = try JSONDecoder()
        .decode(User.self, from: data)
    // iterate in key order
}

// Atomic increment — returns new value
let views = try await store.increment(
    Key("stats", "views"), by: 1
)

// Transactions — snapshot isolation
try await store.transaction { tx in
    let bal = try await tx.get(
        Int.self, forKey: balanceKey
    )
    try await tx.set(
        (bal ?? 0) - amount,
        forKey: balanceKey
    )
    try await tx.set(
        receipt, forKey: receiptKey
    )
}

Store protocol: get, set (with optional TTL), delete, exists, scan, increment, transaction. Codable convenience extensions for type-safe serialisation. Key is Comparable and Hashable with hasPrefix and appending. Short bounded transactions, no blocking I/O in closures, idempotent mutations. InMemoryStore for dev; FoundationDB for production.

Stage PRD

hosting · operations

Overview

Stage is the hosting and operations product in the Allegro ecosystem. It exists to run workloads produced by Score and other Swift executables without coupling the infrastructure layer to Score internals.

Product Goals

  • Host static output cleanly
  • Host runtime binaries cleanly
  • Treat Swift workloads as a first-class target
  • Remain usable without requiring Score

Boundary with Score

Score builds applications. Stage hosts them.

  • Stage infrastructure must remain Score-independent.
  • Stage may use Score for its own web surfaces.
  • Stage should not require privileged knowledge of Score internals.

Remote Build Compatibility

Stage is the remote build executor for production deploys:

  • Run remote builds using canonical score build semantics
  • Pin Score and Swift toolchain versions per build
  • Use reproducible build environments
  • Return deployment IDs and service URLs after activation
  • Keep artifacts portable beyond Stage

Expected Capabilities

  • Local development and preview workflows
  • Deployment and hosting flows
  • Logs and environment management
  • Support for both static and runtime workloads

The exact CLI and dashboard surface should be documented once implementation begins, not before.

Non-Goals

  • Becoming a general JavaScript hosting platform
  • Forcing users into Score to use Stage
  • Recreating mature infrastructure primitives without clear benefit

Composer PRD

native visual editor · macOS + iPad

Overview

Composer is a native macOS and iPad application for editing canonical Score Swift projects with local preview. Production builds run remotely on Stage so deploy output is deterministic and reproducible.

Composer is intentionally thin: it edits canonical Score projects, orchestrates CLI workflows, and does not introduce a proprietary runtime or project format.

Product Goals

  • Visual-first onboarding to Score
  • Static-first builds by default
  • Deterministic, reproducible production builds
  • Atomic zero-downtime deploys through Stage
  • Portable build artifacts deployable beyond Stage

Single-Worker Model

Composer runs exactly one sandboxed worker at a time: one hydrated project, inactive projects compressed and evicted, no concurrent build containers.

Composer depends on Score CLI as its execution backbone. All three commands (score dev, score build, score deploy) must expose structured diagnostics for editor display and keep their error schema backward-compatible.

Non-Goals

  • Composer-specific runtime
  • Visual-only logic engine
  • Runtime graph interpreter inside the editor
  • Platform-locked project format
  • Hidden infrastructure coupling

Risks and Mitigations

Risk Mitigation
Toolchain drift Pin Score + Swift versions in manifest
iPad toolchain limits Remote production build on Stage
Compile latency Incremental build in score dev
Large project size Compression + eviction policy
Debug complexity Structured logs/diagnostics from CLI
